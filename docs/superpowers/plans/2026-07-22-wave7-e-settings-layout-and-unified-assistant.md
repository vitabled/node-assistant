# Волна 7, План E — раскладка настроек и единый ассистент (MCP × Ассистент)

> Пункт пользователя 9, три части: (1) вкладки настроек не влезают в экран; (2) «чем отличается MCP от
> Ассистента?»; (3) «можно сделать единый интерфейс, чтобы с ассистента управлять и Remnawave, и Node
> Assistant?».

## Часть 1 — вкладки настроек

### Разведка

`frontend/src/components/Settings.tsx:791` — **одиннадцать** вкладок:

```
remnawave | defaults | optimization | monitoring | testservers | mcp | assistant | tokens | transfer | infra | theme
```

Рендер — `:819-825`: `<div className="seg" style={{ width: "fit-content" }}>`. `.seg` — сегментный
контрол в один ряд; `fit-content` не даёт ему переноситься, поэтому на любом реалистичном экране
одиннадцать подписей (в т.ч. «Сервера для тестирования», «Деплой (умолчания)») уезжают за край.

### Развилка

- **(а, по умолчанию)** разрешить перенос: `flex-wrap` + `width:auto`, `.seg` в несколько рядов.
  Минимальная правка, ноль новой навигации, ровно то, что просит пользователь («разместить в
  несколько рядов»).
- (б) сгруппировать вкладки в разделы (вертикальный список слева, как в системных настройках).
  Красивее и масштабируемее, но это перестройка экрана, которую не просили, и она конфликтует с
  мобильной раскладкой (`@media(max-width:820px)`), где `.seg` уже скроллится.

### Ф1 — перенос вкладок → verify: `npm test` + скриншоты

- `.seg` в `index.css` не трогать (он используется ещё в ~6 местах — правило хирургических правок).
  Добавить модификатор `.seg-wrap` (`flex-wrap: wrap; width: auto`) и повесить его только на панель
  вкладок настроек.
- Скругления сегментного контрола рассчитаны на один ряд (первый/последний элемент) — при переносе
  середина ряда получит прямые углы. Задать радиусы на **каждый** ряд нельзя без JS, поэтому в
  `.seg-wrap` перейти на индивидуальные скруглённые «пилюли» с зазором — визуально ближе к
  apple-скину, где сегментный контрол и так пилюльный.
- Проверить на всех трёх скинах (apple/console/neon) и в обеих темах; на ≤820px убедиться, что
  горизонтальный скролл не вернулся другим путём.
- Тест `Settings.test.tsx`: все 11 подписей отрендерены; клик по последней («Тема») переключает
  контент (то есть до неё физически можно дотянуться).

## Часть 2 — чем MCP отличается от Ассистента (ответ)

Проверено по коду; это два **противоположных конца** одной идеи, и сегодня они не связаны.

| | **MCP** | **Ассистент** |
|---|---|---|
| Кто мы в этой схеме | **сервер** — отдаём инструменты наружу | **клиент** — сами ходим в LLM |
| Где живёт | контейнер `node-installer-mcp`, оркестрируется через DooD (`services/mcp_server.py`) | внутри бэкенда, `services/ai_agent.py` |
| Кто потребитель | внешний MCP-хост: Claude Desktop, Cursor, IDE | наш собственный чат во вкладке «Ассистент» |
| Инструменты | **156** (smoke-проверка): полный API Remnawave через `@remnawave/backend-contract` 2.9.14 + 9 read-only ручек в наш backend (`mcp/src/tools/node-assistant.ts`) | **4** read-only: `list_rules`, `list_subscriptions`, `node_health`, `list_nodes` (`services/ai_agent.py:90-160`) |
| Аутентификация | `Bearer MCP_AUTH_TOKEN` (Fernet-волт), внутрь — managed readonly API-токен `nai_…` | ключ провайдера в Fernet-волте, наружу к LLM |

Практический вывод: **сегодня ассистент не умеет управлять Remnawave.** У него четыре
read-only инструмента, и ни один не про панель, кроме перечисления нод. Всё богатство (156
инструментов) доступно только внешнему клиенту через MCP — то есть человеку, который поставил Claude
Desktop, а не пользователю нашей панели.

Ответ на вопрос «можно сделать единый интерфейс?» — **да, и правильный способ именно один**: научить
наш `ai_agent` быть MCP-**клиентом** нашего же MCP-сервера. Тогда тот самый набор инструментов, что мы
уже отдаём наружу, становится доступен внутреннему чату, и второй раз описывать контракт Remnawave не
придётся.

## Часть 3 — единый ассистент

### Развилки

**Р1. Как ассистент получает инструменты Remnawave.**
- **(а, по умолчанию)** `ai_agent` подключается к `node-installer-mcp` по Streamable HTTP как MCP-клиент.
  Транспорт уже реализован и проверен (`mcp/src/index.ts`, `smoke.mjs`: initialize + tools/list по HTTP,
  403 без токена). Один интеграционный шов, ноль дублирования контракта.
- (б) дописать десяток in-process инструментов в `ai_agent`. Меньше движущихся частей, но это ручное
  повторение куска `@remnawave/backend-contract`, который мы уже вендорим, — и он разъедется при
  следующем бампе.

**Р2 — БЛОКЕР, требующий решения до кода.** MCP-контейнер сейчас **один общий на всю инсталляцию** и
несёт креды **последнего включившего аккаунта** (зафиксировано в CLAUDE.md §8c как известное
ограничение). Для внешнего клиента это терпимо: включает оператор, он же и потребитель. Для
внутреннего чата — нет: аккаунт Б задал бы вопрос и получил бы данные панели аккаунта А. Варианты:

- **(а, по умолчанию) гейт по владельцу.** Инструменты Remnawave появляются у ассистента **только**
  если текущий аккаунт — тот, чьи креды сейчас в контейнере (`mcp_server` уже знает, кто его
  запускал; сохранить `owner_account_id` в конфиге). Остальным — честная плашка «MCP запущен другим
  аккаунтом». Дёшево, безопасно, не врёт.
- (б) контейнер на аккаунт. Честно изолировано, но это N контейнеров, N портов и оркестрация
  жизненного цикла — отдельная волна.
- (в) прокидывать креды аккаунта в каждый вызов. Требует правок в форке MCP (он берёт креды из env
  на старте) — самый дорогой путь.

Беру **(а)**: пункт про изоляцию из этого плана не выкидывается, а становится явным и видимым в UI.

**Р3. Права.** MCP умеет `readonly` (env `REMNAWAVE_READONLY` регистрирует только read-инструменты).
По умолчанию ассистент подключается **в readonly**, а мутирующие инструменты включаются отдельным
тумблером с предупреждением. Причина та же, что у правил автоматизации: действия деструктивны, а
`AiConfig.readonly` уже существует как поле — используем его, а не заводим второе.

### Ф2 — MCP-клиент в `ai_agent` → verify: `cd backend && python -m pytest`

- `services/mcp_client.py` (NEW): минимальный Streamable-HTTP клиент — `initialize`, `tools/list`,
  `tools/call`. Без SDK: транспорт — обычный POST с сессионным заголовком, как в `mcp/smoke.mjs`.
  Таймауты, **никаких ретраев на 401/403** и никакого логирования тела (там уходят токены панели).
- `mcp_server.py` — `owner_account_id` в `McpConfig` (пишется при `start`), `status()` его отдаёт.
- `ai_agent`: если MCP включён, доступен и принадлежит текущему аккаунту — `tools/list` подмешивается
  к четырём встроенным. Имена префиксуются (`mcp__…`), чтобы не столкнуться со встроенными.
  Кэш списка инструментов на несколько минут: `tools/list` на каждый тёрн — лишний round-trip.
- Деградация: MCP выключен/недоступен/чужой → ассистент работает как сегодня, без ошибки в чате.
- Тесты `backend/tests/test_mcp_client.py`: список инструментов подмешивается; чужой аккаунт их не
  видит; недоступный MCP не роняет `chat`; 401 не ретраится.

### Ф3 — UI единого ассистента → verify: `tsc --noEmit` + `npm test`

- `automation/AiChat.tsx` — плашка над композером: «Инструменты: базовые (4)» либо
  «Инструменты: базовые + Remnawave (N)», с причиной, когда Remnawave недоступен («MCP выключен» /
  «MCP запущен другим аккаунтом»). Пользователь должен понимать границы возможностей **до** вопроса,
  а не из отказа в ответе.
- `settings/AiSettingsTab.tsx` — тумблер «Разрешить изменяющие действия Remnawave» (по умолчанию
  выключен) + ссылка на вкладку MCP.
- `settings/McpTab.tsx` — строка «Ассистент использует этот MCP: да/нет» и явное указание владельца.
  Это и есть место, где ограничение из Р2 перестаёт быть скрытым.
- Тесты: плашка отражает три состояния; тумблер мутаций не показывается, когда Remnawave-инструментов нет.

## Риски

- **Разрастание контекста.** 156 инструментов в один промпт — это десятки килобайт схем на каждый
  тёрн и дороже, и хуже по качеству выбора. Обязательно фильтровать набор (в Ф2 — хотя бы по
  read/write и по префиксу домена), а не подмешивать всё подряд. Замерить размер промпта до/после.
- **Ассистент получает право менять прод.** Ф3 включает мутации только явным тумблером; в чате
  действие, меняющее состояние, должно быть видно как `tool_call` в логе (уже так: события
  `tool_call`/`tool_result` стримятся ndjson).
- **`.seg` используется не только в настройках** — модификатор, а не правка базового класса.
- **Не трогать `mcp/`-форк.** Ф2 — только клиент на нашей стороне; изменения в форке потянут
  ре-верификацию 156 инструментов.

## Критерий готовности

1. Все 11 вкладок настроек видны без горизонтальной прокрутки на десктопе; мобильная раскладка не сломана.
2. В CLAUDE.md появился короткий раздел «MCP vs Ассистент» с таблицей выше (ответ зафиксирован в доке,
   а не только в чате).
3. Ассистент видит инструменты Remnawave, когда MCP включён и принадлежит аккаунту; иначе честно
   сообщает почему.
4. Мутирующие действия выключены по умолчанию.
