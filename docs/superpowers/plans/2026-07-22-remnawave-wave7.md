## Пункт 4 — ответ и решение

Вопрос: *«Всё так же страница с импортом HTML, хотя мы меняли логику, чтобы менять весь `/frontend`
целиком. Как это работает сейчас? Почему нет редактора страницы по умолчанию?»*

**Как работает сейчас.** Из семи фаз Плана E Волны 6 отгружена **только Ф1** — починка сломанного
деплоя (`f8a82a8`). Она не добавляла редактор; она чинила то, что деплой страницы подписок вообще не
работал: мы монтировали произвольный `index.html` поверх EJS-шаблона собранной Vite/React-SPA и тихо
убивали `<%- panelData %>` — единственный канал данных страницы, а с пустым `REMNAWAVE_API_TOKEN`
контейнер завершался кодом 1. Механика монтирования осталась прежней:
`backend/app/services/panel_pipeline.py:302-309` эмитит
`- ./index.html:/opt/app/frontend/index.html`, и только когда `subpage_html` непустой.

**Почему нет редактора страницы по умолчанию.** Чтобы редактировать «страницу по умолчанию», её надо
сначала откуда-то взять — а лежит она внутри образа (`/opt/app/frontend`, 160 файлов, 6.9 МБ). Именно
это и есть Ф4 Плана E («стягивание дерева с живого контейнера + кэш по digest») плюс Ф7 («файловый
браузер/редактор»). Ни одна из них не реализована.

**Решение:** новый план не пишу — `docs/superpowers/plans/2026-07-21-wave6-e-subscription-page-frontend.md`
проверен на живом образе 7.2.6 и остаётся верным. Он **переносится в Волну 7 с повышенным приоритетом**
как «План G», фазы Ф2–Ф7 без изменений. Дублировать его содержимое здесь было бы копией, которая
разъедется с оригиналом.

## Порядок

Планы независимы; последовательность — по соотношению «раздражение пользователя / цена».

1. **A** — три дефекта, видимых на каждом экране, стоят несколько часов. Сначала они.
2. **C** — селекторы панели: разблокирует работу с несколькими панелями, которую реестр уже умеет.
3. **D** — хостинги: пять отдельных улучшений, любое отгружаемо в одиночку.
4. **G** (= Волна 6, План E Ф2–Ф7) — редактор страницы подписок. Самый большой кусок, но пользователь
   спрашивает о нём второй раз, и премисса уже проверена живьём.
5. **B** — импорт из подписки. Зависит от Плана A (резолв страны) и требует решения по развилке
   «домен или адрес».
6. **E** — раскладка вкладок сразу; единый ассистент — после того, как решён вопрос владельца MCP.
7. **F** — CLIProxyAPI через OAuth. Самый рискованный (открытый прокси, бан IP), поэтому последним,
   с наибольшим запасом внимания.

## Сквозные развилки, требующие решения пользователя

Решения по умолчанию проставлены в планах; здесь — только те, где цена ошибки высока.

- **B/Р1 — домен или адрес.** Ссылки подписки несут домен, а `servers.ip` валидируется как IPv4.
  По умолчанию резолвим при импорте и пиним адрес (меньше правок). Честнее — хранить домен и
  резолвить при пробе, но это меняет модель. Если ноды за CDN/round-robin — нужен второй вариант.
- **E/Р2 — владелец MCP.** Контейнер один на инсталляцию и несёт креды последнего включившего
  аккаунта. Пока это так, инструменты Remnawave в ассистенте видит только владелец. Альтернатива
  (контейнер на аккаунт) — отдельная волна.
- **F/Р1 — том вместо bind-mount.** Бэкенд управляет docker через сокет, поэтому пути в `docker run` —
  хостовые. Именованный том снимает проблему трансляции путей; конфиг засевается **до** старта,
  иначе существует окно, в котором прокси открыт всему миру.
- **D/Р1 — детализация карты против веса бандла.** `countries-50m` рисует границы лучше, но
  учетверяет географию, а перф-база Волны 6 уже намерила 1.43 МБ JS. По умолчанию чиним обводкой.

## Что явно НЕ входит в волну

- Слияние `panel_jobs` (SSH-креды панелей) в серверный реестр — нарушило бы правило «секреты целевых
  серверов только на клиенте».
- Переезд инфра-биллинга на SVG-флаги (`getFlagEmoji` остаётся ради `InfraProviders.tsx:78`).
- Правки в форке `mcp/` — потянут ре-верификацию 156 инструментов.
- Незакрытый долг Волны 6: **D целиком** (eGames-вариант панели) и **F Ф2–Ф7** (лог-шторм, дэшборд,
  контраст, бандл). Не отменены, просто не в этой волне.
- Задача «полный ревью кода и оптимизация после Волны 6» — остаётся отдельной, после этой волны.

## Общая верификация

Node на машине разработки нет. Бэкенд: `cd backend && python -m pytest`. Фронтенд — в Docker:
`docker build -f frontend/Dockerfile --target builder -t ni-frontend-test frontend`, затем
`docker run --rm -v <abs>/frontend:/w -w /w ni-frontend-test npx vitest run` и `npx --no-install tsc --noEmit`
(в Git Bash — `MSYS_NO_PATHCONV=1`). Playwright-харнессы — образ `mcr.microsoft.com/playwright:v1.61.1-jammy`
(версия пиннится репозиторием).
