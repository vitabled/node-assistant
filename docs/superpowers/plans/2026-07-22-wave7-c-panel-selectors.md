# Волна 7, План C — селекторы панели Remnawave (конфиги + установка)

> Пункты пользователя 5 и 6. Оба про одно и то же: реестр панелей из Волны 5 (План K) существует, но
> им пользуется ровно один экран. Раздел «Пользовательские конфиги» молча ходит в активную панель,
> а «Установка» про реестр не знает вовсе.

## Разведка (проверено по коду)

**Реестр панелей уже есть и спроектирован под это.** `backend/app/models/settings.py:14-27`:

```
class PanelEntry:  id, name, kind ("custom" | "deployed"), panel_url, api_token,
                   default_internal_squad_ids, default_external_squad_ids
class RemnawaveRegistry: panels: list[PanelEntry], active_panel_id: str
```

`kind="deployed"` заведён, но **нигде не выставляется** — задел под ровно тот сценарий, который просит
пункт 6.

Ключевой трюк Волны 5 — `AppSettings._resolve_active_panel` (`models/settings.py:134-165`,
`mode="after"`): поле `.remnawave` не хранится, а **вычисляется** проекцией активной записи реестра.
Поэтому ~13 мест, читающих `.remnawave`, работают на активной панели и их не трогали. Legacy-конфиг
одной панели автомигрируется в реестр с id `"primary"` (`:145-151`), битый `active_panel_id` падает на
первую (`:155-158`).

Роуты реестра: `GET/POST /api/settings/remnawave/panels`, `PUT/DELETE /panels/{id}`,
`POST /panels/{id}/activate` (см. CLAUDE.md §5). UI — `Settings.tsx:122-147` (`PanelSelector`),
подключён на `:242`.

### Пункт 5 — конфиги ходят в активную панель молча

`backend/app/api/config_templates.py:23-27`:

```python
def _client() -> RemnavaveClient:
    cfg = AppSettings(**storage.load_settings()).remnawave
    if not cfg.panel_url or not cfg.api_token:
        raise HTTPException(400, "Remnawave не настроен")
    return RemnavaveClient(cfg.panel_url, cfg.api_token)
```

Один резолвер на три ручки: `POST /{id}/export` (`:68-85`), `GET /import/panel` (`:88-94`),
`POST /import/panel/{uuid}` (`:97-104`). Выбора нет — всегда активная. Именно это и просят изменить,
сохранив активную как значение по умолчанию.

### Пункт 6 — «Установка» не знает про реестр

Раздел установки панели работает с `panel_jobs_<account>` в **localStorage** (`rw/PanelDashboard.tsx`,
CLAUDE.md §7c): там лежат SSH-креды развёрнутых панелей, и лежат они на клиенте **намеренно** —
секреты целевого сервера мы не храним. Реестр же (`settings.json`) хранит `panel_url` + `api_token`
панели, то есть это два разных хранилища с разным содержимым и разной политикой.

## Развилки

**Р1. Как «добавить существующую панель» в «Установке».**
- **(а, по умолчанию)** показать в «Установке» тот же реестр из настроек (`/api/settings/remnawave/panels`)
  с добавлением/активацией/удалением. «Основная» — одно поле `active_panel_id` на обоих экранах, что
  буквально и просит пользователь: *«выбранная панель по умолчанию в этой вкладке выбирается основной
  и в настройках»*.
- (б) слить `panel_jobs` в реестр. Отвергаю: в `panel_jobs` лежат SSH-креды, которые мы сознательно
  держим только в браузере; перенос их в `settings.json` — это хранение чужих секретов на сервере,
  прямо против правила проекта. Связь «развёрнутая панель ↔ запись реестра» делаем **мягкой**
  (см. Ф3): кнопка «Добавить в реестр» на карточке развёрнутой панели, которая заводит запись с
  `kind="deployed"` и предзаполненным `panel_url`, а токен оператор вводит сам.

**Р2. Как конфиги выбирают панель.**
- **(а, по умолчанию)** необязательный query/body-параметр `panel_id`; пусто → активная панель.
  Обратная совместимость полная, старые вызовы не ломаются.
- (б) обязательный параметр — сломает существующий фронт и не даст «по умолчанию — панель по умолчанию».

**Р3. Где живёт резолвер `panel_id → RemnavaveClient`.** Такой резолвер понадобится и другим модулям
(миграция, синк, ассистент). Кладу его **один раз** в `services/panel_registry.py` и переиспользую,
а не копирую `_client()` по роутерам.

## Риски

- **`_resolve_active_panel` — валидатор модели, а не хранилище.** Любая запись `settings.json` должна
  идти через `AppSettings`, иначе `active_panel_id` можно выставить на несуществующий id и получить
  тихий фолбэк на первую панель. Проверить это тестом при добавлении записи из «Установки».
- **Удаление активной панели.** `Settings.tsx:144` блокирует удаление только когда панель одна.
  Удаление именно активной оставляет битый `active_panel_id`, который спасает фолбэк на первую —
  поведение рабочее, но неочевидное. В Ф4 показать это в UI («главной станет: …»), логику не менять.
- **Токен панели — plaintext в `settings.json`** (статус-кво, зафиксирован в CLAUDE.md §5). Новый
  экран не должен показывать токен: в списке — только имя и URL, как сейчас в настройках.

## Критерий готовности

1. В «Пользовательских конфигах» есть селектор панели; без выбора работает как раньше (активная).
2. В «Установке» виден тот же реестр; «сделать главной» там и в настройках меняет одно и то же поле.
3. Импорт с чужим `panel_id` даёт 404, а не молча тянет из активной.
4. `pytest` и `vitest` зелёные; `PanelSelector` существует в единственном экземпляре.
