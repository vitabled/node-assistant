# Панель node-installer: аккаунты, авторизация, UI-оверхол, мульти-подписки, хосты

## Контекст

Строим крупный набор изменений панели node-installer (FastAPI async + React18/TS/Vite/Tailwind SPA). Всё продиктовано владельцем проекта в одной сессии и разбито здесь на самодостаточные фазы для пофазного исполнения.

Смысловые блоки: (1) авторизация + мульти-аккаунты в стиле Google с полной per-account изоляцией данных; (2) фикс бага восстановления текущей страницы после перезагрузки; (3) светлая/тёмная/системная тема + перевод захардкоженных компонентов на CSS-переменные + реорганизация лейаута; (4) переработка формы деплоя (секции, сворачивание, перенос полей, whitelist fail2ban/firewall, cert-провайдеры, флаги); (5) реструктуризация этапов деплоя + управление отдельными шагами после деплоя + отслеживание срока сертификата; (6) мульти-подписки Xray-Checker через отдельный контейнер локального сервера подписок; (7) «Управление SSL» (деплой серта на ноду); (8) раздел «Хосты» (локальные шаблоны хоста Remnawave).

**Ключевые решения (уже приняты владельцем, не переоткрывать):**
- **Аккаунты:** вход в панель = вход в аккаунт. Регистрация создаёт пустой аккаунт (логин уникален, логин+пароль, авто-вход после регистрации). Кнопка «Сгенерировать пароль» на регистрации. Свитчер показывает только добавленные на этом устройстве аккаунты + «Добавить аккаунт». Переключение мгновенное без пароля. Сессия персистентная (переживает перезагрузку и перезапуск браузера). Кнопки «Выйти из аккаунта» и «Удалить аккаунт с устройства». Пароли — солёный хэш (bcrypt/argon2 через passlib).
- **Изоляция:** у каждого аккаунта своя вселенная — server-side под `DATA_DIR/accounts/<id>/…` (`settings.json`, `templates.json`, `traffic_rules.json`, `infra_billing.db`, метрики), а `deploy_jobs` в localStorage — с суффиксом аккаунта. Все роуты резолвят путь по активному аккаунту из сессии.
- **PIN-гейт infra-billing удаляется полностью** (код, роуты `/api/infra-billing/auth/*`, заголовок `X-Billing-Session`, фронт-гейт, поле PIN в настройках биллинга) — панель уже под аккаунтом.
- **Тема:** режим Светлая/Тёмная/Системная (Системная = `prefers-color-scheme`), дефолт Системная, выбор per-account, в новой вкладке «Тема» в Настройках. Захардкоженные тёмные Tailwind-компоненты (`DeployForm`, `DeployCard`, `StepProgress`, `CertsForm`, окно деплоя) переводятся на CSS-переменные, чтобы следовать режиму + акценту/плотности (`theme/tweaks.ts`).
- **Checker-подсистема — осознанное исключение из per-account изоляции:** один общий контейнер xray-checker + один общий контейнер локального сервера подписок проверяют агрегат подписок ВСЕХ аккаунтов. Каждая позиция агрегата тегируется владельцем (аккаунт + исходная подписка). Метрик-стор общий, но каждый sample тегируется аккаунтом; дашборд аккаунта показывает только его подписки/метрики/историю (фильтрация на выдаче). Всё остальное (settings/шаблоны/traffic-rules/deploy-задачи/infra-billing/хосты) — строго изолировано.
- **Cert-провайдеры:** Cloudflare (DNS-01, acme.sh), Let's Encrypt (HTTP-01), ZeroSSL (acme.sh + EAB). Поле «Cloudflare API токен» показывается только при выборе Cloudflare.
- **Флаги стран:** SVG-набор `flag-icons` вместо ASCII-эмодзи.

**Текущее состояние (точки входа в код):**
- Данные: JSON под `DATA_DIR` (`storage.py`), infra-billing SQLite (`services/infra_billing_store.py`), метрики чекера (`services/metrics_store.py`), deploy-карточки — localStorage `deploy_jobs`.
- Пайплайн деплоя: `services/pipeline.py` (11 шагов, `_begin_step`), метки `services/task_store.py::STEP_LABELS`, фронт-метки `components/StepProgress.tsx::DEPLOY_STEPS`.
- Форма: `components/DeployForm.tsx` (`FormData`, `FORM_DEFAULT`, `validateForm`), модель `models/deploy.py::DeployRequest`.
- Лейаут: `App.tsx` (топбар, поповер «Внешний вид», date-лейбл), `components/Sidebar.tsx` (nav + футер «Remnawave»), тема `theme/tweaks.ts`, `index.css` (CSS-переменные тёмной палитры).
- Xray-Checker: `services/xray_checker.py` (оркестратор контейнера, DooD), `api/xray_checker.py` (poller), `components/Dashboard.tsx` (статус-страница), `components/Settings.tsx` (XrayCheckerSettings).
- Certs: `components/CertsForm.tsx`, `api/certs.py` (RENEW), пункт nav `certs`.
- Remnawave-клиент: `services/remnawave_client.py` (методов hosts НЕТ — раздел «Хосты» локальный).

## Стратегия

Доменный граф (порядок = топосорт):

```
Ф1 backend-аккаунты ─┬─ Ф2 frontend-auth+route ─┬─ Ф3 тема+лейаут
                     │                          ├─ Ф4 форма-деплоя (fe+модель) ─ Ф5 пайплайн-поведение ─ Ф6 шаги-реструктуризация ─ Ф7 управление-шагами+cert-срок
                     │                          ├─ Ф9 дашборд-мультиподписки
                     │                          ├─ Ф10 управление-SSL
                     │                          └─ Ф11 хосты
                     └─ Ф8 контейнер-сервера-подписок ─ Ф9
```

Общие контракты между фазами:
- **Активный аккаунт** резолвится в backend из сессионного токена (Ф1). Все downstream-роуты, читающие/пишущие пользовательские данные, обязаны использовать account-scoped путь (helper из Ф1).
- **Токен аккаунта на устройстве** — заголовок запроса (напр. `X-Account-Session`) + список аккаунтов в localStorage (Ф2). Фронт всех разделов шлёт активный токен.
- **CSS-переменные темы** (Ф3) — палитры светлой/тёмной в `index.css`; downstream-компоненты используют только `var(--…)`, не хардкод.
- **Новые поля DeployRequest** (Ф4) — `whitelist_ips`, `allow_ssh_all`, `install_vnstat`, `install_trafficguard`, `cert_provider`; Ф5 навешивает поведение.
- **Структура шагов** (Ф6): новый `STEP_LABELS` + группы; Ф7 адресует шаги для переустановки/удаления по стабильному ключу.
- **Агрегат подписок** (Ф8): контейнер отдаёт объединённую подписку, каждая позиция тегирована `account:sub` в remark/name; Ф9 фильтрует по активному аккаунту.

Порядок исполнения продиктован зависимостями данных: сначала backend-аккаунты (все изолированные хранилища завязаны на резолв аккаунта), затем frontend-auth (без него нельзя тестировать per-account экраны), дальше домены расходятся.

## Риски

- **Регрессия изоляции:** ошибка резолва аккаунта → утечка данных между аккаунтами. Смягчение: helper единой точкой (Ф1), edge-тест «аккаунт А не видит данные Б» в каждой фазе, работающей с per-account хранилищем.
- **Ломка пайплайна при переиндексации шагов:** `STEP_LABELS`, `_begin_step(index)`, слот HAProxy (index 7) и фронт `DEPLOY_STEPS` жёстко связаны по индексам. Смягчение: Ф6 меняет их одним согласованным изменением + смоук деплой-скриптогенерации.
- **Checker vs изоляция:** общий стор с тегами — единственная точка, где данные аккаунтов физически рядом. Смягчение: фильтрация обязательна во ВСЕХ checker-запросах дашборда; edge-тест на протечку.
- **ZeroSSL EAB:** acme.sh ZeroSSL требует EAB kid/hmac либо регистрацию по email — заложить поля/шаг (Ф5), при первой интеграции записать в CLAUDE.md §6.
- **Deploy-задачи в localStorage:** per-account суффикс должен применяться консистентно во всех местах чтения/записи `deploy_jobs` (Ф2 задаёт контракт; Ф7/Ф9/Ф10 его переиспользуют).
- **Continuous Memory:** каждая фаза обязана обновить CLAUDE.md (§ архитектура) в своём же изменении — иначе reference разъедется с кодом.

## Фаза 1 — Backend: аккаунты, авторизация, per-account изоляция
<!-- circle: status=pending order=10 deps=[] autonomy=auto obstacle="" -->

**Подход:** серверный стор аккаунтов + сессии + резолв активного аккаунта из токена, и неймспейс всех изолированных хранилищ под `DATA_DIR/accounts/<id>/`. Отвергнуто: хранить аккаунты только на клиенте (нельзя изолировать server-side данные) и внешний identity-провайдер (избыточно для однопанельного self-host).

**Шаги:**
1. Зависимость: `passlib[bcrypt]` (проверить актуальную версию через `pip index versions passlib` перед добавлением). Хэш пароля — bcrypt, соль встроена.
2. `services/accounts_store.py` — SQLite `DATA_DIR/accounts.db`: `accounts(id, username UNIQUE, password_hash, created_at)`, `sessions(token, account_id, created_at)`. Методы: `create_account(username, password)` (409 при занятом логине), `verify(username, password)`, `issue_session(account_id)`, `resolve(token) -> account_id | None`, `delete_session(token)`. Токен — `secrets.token_urlsafe`, персистентный (без TTL — сессия живёт как в Google).
3. Резолв активного аккаунта: FastAPI dependency `current_account(x_account_session: Header)` → `account_id` или 401. Подключить ко всем роутам, читающим/пишущим пользовательские данные.
4. Account-scoped пути: helper `account_dir(account_id) -> DATA_DIR/accounts/<id>/`. Переключить `storage.py` (settings/templates/traffic_rules), `infra_billing_store.py`, `metrics_store.py` (кроме checker-агрегата — см. Ф8/Ф9, там общий стор с тегом) на построение пути от активного аккаунта. Отсутствует директория/файл → создать с дефолтами (пустой аккаунт).
5. Роуты: `POST /api/accounts/register` (создать+авто-сессия→токен), `POST /api/accounts/login` (→токен), `POST /api/accounts/logout` (удалить сессию), `GET /api/accounts/me` (проверка токена→username). Генератор пароля — на клиенте (Ф2); сервер только принимает.
6. **Удалить PIN-гейт infra-billing:** роуты `/api/infra-billing/auth/*`, проверку заголовка `X-Billing-Session`, поле PIN в `billing_settings`/модели настроек биллинга. `grep` по `X-Billing-Session`, `verify-session`, `billing.*pin` — убрать все следы.

**Edge-cases:** дубликат логина (409); неверный пароль (401); пустые/whitespace логин/пароль (422); запрос к защищённому роуту без/с невалидным токеном (401); удалённая на сервере сессия при живом токене на клиенте (401 → клиент разлогинит, Ф2); одновременная регистрация одинакового логина (UNIQUE constraint ловит гонку); аккаунт без директории (лениво создаётся); malformed токен (401, без 500).

**Verify-гейт (исполняемый смоук):** поднять backend, `curl` сценарий: register → получить токен → `GET /api/settings` с токеном (200, дефолты) → тот же GET без токена (401) → register с тем же логином (409) → login с неверным паролем (401). Плюс проверить, что settings двух разных аккаунтов пишутся в разные `accounts/<id>/settings.json`.

**Контракт для следующих фаз:** заголовок `X-Account-Session`; dependency `current_account`; helper `account_dir`. Роуты аккаунтов как выше. Следующий шаг: Ф2 строит фронт поверх этих роутов.

## Фаза 2 — Frontend: экран входа, свитчер аккаунтов, восстановление роута
<!-- circle: status=pending order=20 deps=[1] autonomy=auto obstacle="" -->

**Подход:** гейт всего SPA за экраном входа; список аккаунтов устройства + их токены в localStorage; активный токен шлётся во все запросы. Восстановление активной вкладки — из per-account localStorage. Отвергнуто: хранить один токен (не даёт мгновенного переключения) и держать роут в URL-хэше (per-account требование проще через localStorage-ключ с суффиксом аккаунта).

**Шаги:**
1. `auth/session.ts`: модель устройства `{accounts: [{id, username, token}], activeId}` в localStorage. Хелперы add/remove/switch/getActiveToken. Централизованный `fetch`-врапер, добавляющий `X-Account-Session: <activeToken>`; на 401 — сбросить активную сессию и показать вход.
2. Экраны: `Login` (логин+пароль, кнопка «Регистрация»), `Register` (логин+пароль+кнопка «Сгенерировать пароль» — стойкий случайный пароль в поле с возможностью скопировать; после успеха — авто-вход). Экран по умолчанию — вход.
3. Свитчер аккаунтов (Google-стиль): показывает только аккаунты устройства + «Добавить аккаунт» (→ экран входа). Переключение мгновенное (смена activeId, без пароля). Кнопки «Выйти из аккаунта» (logout + убрать активную сессию, оставить в списке устройства как разлогиненный или убрать — см. default ниже) и «Удалить аккаунт с устройства» (убрать из списка; серверные данные не трогаем).
   - Default: «Выйти из аккаунта» удаляет серверную сессию (logout) и убирает аккаунт из активных, но оставляет запись логина в свитчере со статусом «нужен вход». Если при имплементации это усложняет UX свитчера — fallback: «Выйти» полностью убирает аккаунт из свитчера (эквивалент «удалить с устройства» + серверный logout).
4. Гейт в `App.tsx`: пока нет активного валидного токена (`GET /api/accounts/me` ок) — рендерим экран входа/свитчер, не SPA.
5. **Фикс бага восстановления страницы:** активный `Tab` сохранять в localStorage per-account (ключ `active_tab:<accountId>`) при каждой смене; при загрузке — восстанавливать. Дефолт при отсутствии — `dashboard`. `deploy_jobs` — переключить на per-account ключ `deploy_jobs:<accountId>` во ВСЕХ местах чтения/записи (`DeployDashboard` и др.) — это общий контракт для Ф7/Ф9/Ф10.

**Edge-cases:** активный токен протух на сервере (401 → врапер разлогинивает, экран входа); переключение на аккаунт, чья сессия удалена (повторный вход); удаление активного аккаунта из свитчера (переключиться на другой или экран входа); пустой список аккаунтов (только экран входа); F5/перезагрузка (восстановление вкладки — happy path бага); back/forward и переключение вкладок браузера (сессия сохраняется); сгенерированный пароль скопирован но не сохранён пользователем (показать явно до ухода с экрана).

**Verify-гейт:** headless-playwright: открыть SPA → редирект на вход → регистрация нового аккаунта → попасть в SPA (dashboard) → сменить вкладку на «Настройки» → F5 → остаться на «Настройки» (маркер в DOM). Добавить второй аккаунт, переключиться, убедиться что вкладка/данные свои. Console clean, документ+bundle 2xx.

**Контракт:** localStorage-ключи `deploy_jobs:<accountId>`, `active_tab:<accountId>`; fetch-врапер с `X-Account-Session`. Следующий шаг: остальные фазы используют врапер и per-account ключи.

## Фаза 4 — Форма деплоя: реорганизация, сворачивание, перенос полей, whitelist, cert-провайдер, флаги
<!-- circle: status=pending order=40 deps=[3] autonomy=auto obstacle="" -->

**Подход:** реструктурировать `DeployForm` (секции/сворачивание/перенос), добавить новые поля в `FormData` и `DeployRequest` (модель принимает, поведение навесит Ф5), заменить флаги на SVG. Отвергнуто: делать поведенческие изменения пайплайна здесь же (разнесено в Ф5 ради размера фазы и чистого контракта).

**Шаги (порядок секций в режиме Remnanode):**
1. Табы режима — как есть.
2. **Сервер** — перенести сюда тумблер «Обновить систему перед стартом» (`update_system`).

## Фаза 11 — Раздел «Хосты»: локальный редактор шаблонов хоста Remnawave
<!-- circle: status=pending order=110 deps=[2] autonomy=auto obstacle="" -->

**Подход:** новый раздел с CRUD локальных шаблонов хоста (per-account, server-side), форма 1:1 повторяет форму хоста Remnawave (по скриншотам владельца). Remnawave API не трогаем — шаблоны задействуются позже при деплое в Remnawave. Отвергнуто: интеграция с Remnawave hosts API (владелец выбрал локальные шаблоны; методов hosts в клиенте нет).

**Шаги:**
1. Nav-пункт «Хосты» (`Sidebar`, `App.tsx` CRUMB, `Tab`-union). Хранилище `hosts.json` в account_dir + CRUD-роуты (session-gated).
2. Форма шаблона хоста — две вкладки, точно по форме Remnawave:
   - **Видимость хоста** — тумблер (общий, вверху).
   - **Вкладка «Основные» → Базовые параметры:** Примечание* (text, required); Инбаунд (селектор/кнопка «Выбрать», дефолт «Инбаунд не выбран»); Адрес* (text) + Порт* (number); Tag (селектор, дефолт `ROUTING_HOST`; подпись «Теги не видны конечным пользователям. Тег будет отправлен только с RAW подпиской.»); Ноды (мультиселект; подпись «влияет только на визуальное отображение»); Исключить из внутренних сквадов (мультиселект).
   - **Вкладка «Расширенные» → Переопределения соединений:** SNI (text); «Переопределить SNI из адреса» (тумблер); «Оставить SNI пустым» (тумблер); Хост (text) + Путь (text); Security Layer (селектор, дефолт «По умолчанию»); ALPN (селектор) + Отпечаток/Fingerprint (селектор); Vless Route ID (number, «1–65535, пусто/0 = выкл»).
   - Далее на «Расширенные»: «Скрыть хост» (тумблер); «Исключить из типа подписки» (чекбоксы: Xray JSON, Xray Base64, Mihomo, Stash, Singbox, Clash).
   - **Xray Json & Raw:** «Шаблон Xray JSON» (селектор); под-конфиги-кнопки: xHTTP, Mux, SockOpt, Final Mask (каждая открывает свой суб-редактор — заложить хранение их полей).
   - **Прочие настройки:** Server Description (text, макс 30); «Перемешать хост» (тумблер); «Разрешить небезопасные» (тумблер).
   - **Настройки для Mihomo:** «Включение x25519mlkem768» (тумблер).
   - Кнопка «Сохранить» (disabled пока не заполнены required: Примечание, Адрес, Порт).
3. Список сохранённых шаблонов хоста (карточки/таблица) + редактирование/удаление.

**Edge-cases:** пустые required (Примечание/Адрес/Порт → Сохранить disabled/ошибка); порт вне 1–65535; Vless Route ID пусто/0 (=выкл) vs вне диапазона; Server Description >30 символов (обрезать/ошибка); «Оставить SNI пустым» + «Переопределить SNI из адреса» одновременно (взаимоисключение или явный приоритет); все типы подписки исключены (предупреждение); дубль имени шаблона; удаление шаблона; переключение аккаунта (свои шаблоны, изоляция).
